## 🏛️ CHOIX DE DESIGN & ARCHITECTURE

### Design Pattern: Dual-Panel Model

#### Décision
Implémenter un modèle où chaque type de nœud spécialisé a son propre panneau UI dédié (MathOpPropertyPanel, GetBBValuePropertyPanel, etc.) plutôt que d'avoir un seul panneau monolithique avec énormes switchs.

#### Justification
```
Avant (Problématique):
RenderProperties() {
    switch(nodeType) {
        case GetBBValue: // 50 lignes
        case SetBBValue: // 50 lignes
        case MathOp: // 100 lignes
        case Branch: // 200 lignes
        // Total: 1000+ lignes en escalade
    }
}

Après (Dual-Panel):
RenderProperties() {
    switch(nodeType) {
        case GetBBValue:
            m_getBBPanel->Render();  // Déléguée
            break;
        case MathOp:
            m_mathOpPanel->Render(); // Déléguée
            break;
    }
}
```

#### Avantages
- ✅ **Maintenabilité:** Chaque panel responsable de son propre rendu
- ✅ **Testabilité:** Panels testables indépendamment
- ✅ **Extensibilité:** Ajouter un nouveau type = créer un nouveau panel
- ✅ **Responsabilité Unique:** Pas de géant monolithique
- ✅ **Réutilisabilité:** MathOpPropertyPanel peut être utilisé ailleurs

#### Trade-offs
- ❌ Plus de fichiers (8 vs 2)
- ❌ Allocation mémoire pour chaque panel
- ✅ Mais: Gain en clarté justifie les inconvénients

### Design Pattern: Cascade Synchronization

#### Décision
Implémenter une synchronisation multi-niveaux: UI → Panel → EditorPanel → Template → Executor

#### Architecture
```
User Input (ImGui)
       ↓
Panel.SetBBKey(newValue)
       ↓
Panel.IsDirty() = true
       ↓
EditorPanel.RenderProperties() détecte IsDirty()
       ↓
EditorPanel.m_template.Nodes[i].BBKey = newValue
       ↓
EditNodePropertyCommand créée (undo/redo)
       ↓
m_dirty = true (canvas refresh)
       ↓
RebuildLinks() / SyncCanvasFromTemplate()
       ↓
VSGraphExecutor utilise le BBKey mis à jour
```

#### Justification
- **Atomic Updates:** Chaque changement est une opération complète
- **Reversible:** Undo/Redo intégré naturellement
- **Traceable:** Chaque étape bien définie
- **Debuggable:** Points de break clairs

#### Implémentation
```cpp
// Dans RenderProperties:
if (m_getBBPanel->IsDirty()) {
    // 1. Récupérer la nouvelle valeur
    const std::string oldKey = def.BBKey;
    def.BBKey = m_getBBPanel->GetBBKey();

    // 2. Mettre à jour le template
    for (size_t i = 0; i < m_template.Nodes.size(); ++i) {
        if (m_template.Nodes[i].NodeID == m_selectedNodeID) {
            m_template.Nodes[i].BBKey = def.BBKey;
            break;
        }
    }

    // 3. Créer la commande undo
    m_undoStack.PushCommand(
        std::unique_ptr<ICommand>(new EditNodePropertyCommand(...)),
        m_template);

    // 4. Déclencher le refresh
    m_dirty = true;
    m_getBBPanel->ClearDirty();
}
```

### Choix Technique: Mode Trimodal pour les Opérandes MathOp

#### Décision
Implémenter 3 modes pour chaque opérande: **Variable**, **Const**, **Pin**

#### Modes Détaillés

| Mode | Usage | Exemple | Cas d'Usage |
|------|-------|---------|-----------|
| **Variable** | Référencer une variable blackboard | "playerHealth" | Lire des données dynamiques du BB |
| **Const** | Valeur constante en dur | "42" ou "3.14" | Calculs fixes (scalers, offsets) |
| **Pin** | Utiliser un pin d'un autre nœud | "Node#5:Output" | Connecter les flux de données |

#### Justification
```cpp
struct MathOpOperand {
    enum class Mode { Variable, Const, Pin };
    Mode mode;
    std::string variableName;   // Pour Variable
    std::string constValue;      // Pour Const
    std::string dynamicPinID;    // Pour Pin
};
```

**Flexibilité maximale:**
- Peut combiner les trois modes dans une même opération
- Exemple: `(playerHealth * 2.0) + itemBonus`
  - Left: Variable("playerHealth")
  - Op: "*"
  - Right: Const("2.0")
- Suivi d'une deuxième op avec result → Pin référence

#### Implémentation
```cpp
// GetDisplayString() montre le contexte à l'UI
std::string MathOpOperand::GetDisplayString() const {
    switch (mode) {
        case Mode::Variable: return "[Var] " + variableName;
        case Mode::Const:    return "[Const] " + constValue;
        case Mode::Pin:      return "[Pin] " + dynamicPinID;
    }
}

// Sérialisation préserve le mode
json ToJson() const {
    j["mode"] = static_cast<int>(mode);
    j["var"] = variableName;
    j["const"] = constValue;
    j["pin"] = dynamicPinID;
}
```

#### Trade-offs
- ✅ Très flexible (couvre tous les cas)
- ✅ Simple à étendre
- ❌ Nécessite validation (pas tous les champs utilisés)
- ✅ Mais: Sérialisation JSON préserve tout

### Choix Technique: VariableType::None = "Any Type"

#### Décision
Utiliser `VariableType::None` pour signifier "accepte n'importe quel type" dans la validation des pins.

#### Contexte
```
GetBBValue: 
  - Type réel déterminé par la variable sélectionnée
  - Mais au moment de la création, pas de variable sélectionnée
  - Pin type initialement = None

MathOp Inputs (A, B):
  - Type = Float (fixes)

Connexion GetBBValue → MathOp:
  - Source: None (any type)
  - Target: Float
  - Validation: None est compatible avec Float ✅
```

#### Implémentation
```cpp
// Dans VSGraphVerifier.cpp
if (srcPin->PinType == VariableType::None || 
    dstPin->PinType == VariableType::None) {
    // Accepter la connexion
    continue;  // Pas d'erreur
}

if (srcPin->PinType != dstPin->PinType) {
    // Erreur de type
    AddIssue(r, "E006_DataPinTypeMismatch", ...);
}
```

#### Justification
- **Pragmatique:** Permet une flexibilité maximale
- **Réaliste:** Les types réels se déterminent à l'exécution
- **Sûr:** Executor valide les types au runtime
- **Compatible:** Fonctionne avec les designs existants

### Choix Technique: IsDirty Flag Pattern

#### Décision
Utiliser un pattern `IsDirty/ClearDirty()` pour tracking des changements plutôt que event callbacks.

// Pattern Dirty Flag: Simple & robuste

OnChange.Connect([](){ /* handle change */ });

- ❌ Pas de notification immédiate
- ✅ Mais: Polling par frame accepté

#### Implémentation dans Editor

    // ...
    m_getBBPanel->ClearDirty();  // Reset le flag

### Choix Architectural: DataPins Initialization in AddNode()

#### Décision
Initialiser les DataPins directement dans `AddNode()` plutôt que dans un passe séparé.

#### Implémentation
```cpp
int AddNode(TaskNodeType type, float x, float y) {
    TaskNodeDefinition def;
    def.Type = type;

    // Initialiser les pins ici même
    if (type == TaskNodeType::GetBBValue) {
        DataPinDefinition pin;
        pin.PinName = "Value";
        pin.Dir = DataPinDir::Output;
        pin.PinType = VariableType::None;
        def.DataPins.push_back(pin);
    }
    else if (type == TaskNodeType::MathOp) {
        // Créer pins A, B, Result
    }

    // Reste de l'initialisation...
    m_template.Nodes.push_back(def);
}
```

#### Justification
- ✅ **Atomic:** Node créé avec ses pins en une seule opération
- ✅ **No Orphans:** Impossible d'avoir un node sans pins
- ✅ **Undo-safe:** Command crée le node complet
- ❌ Rend AddNode() plus gros (acceptable)
